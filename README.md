# Events Aggregator API

Сервис-агрегатор мероприятий, который синхронизирует данные из внешнего API провайдера событий и предоставляет интерфейс для управления билетами.

## Основные возможности

- **Автоматическая синхронизация**: Фоновый процесс для периодического получения обновлений о мероприятиях.
- **Управление билетами**: Регистрация пользователей на мероприятия и отмена регистрации по ID билета.
- **Идемпотентность**: Поддержка ключей идемпотентности для предотвращения дубликатов при покупке билетов.
- **Гарантированная доставка уведомлений**: Transactional Outbox паттерн для надёжной отправки уведомлений в Capashino.
- **Сервисный слой для бизнес-логики**: Роутеры остаются тонкими, работа с БД инкапсулирована в сервисах/usecase.
- **Доменные исключения**: Usecase-слой не зависит от HTTP, преобразование в HTTP-ответы выполняется в API-слое.
- **Типобезопасные статусы событий**: Используется `EventStatus(str, Enum)` вместо магических строк.
- **Просмотр мест**: Получение актуального списка доступных мест для конкретного события через API провайдера.
- **Пагинация и фильтрация**: Просмотр списка мероприятий с поддержкой фильтра по дате и постраничной навигации.
- **Здоровье сервиса**: Эндпоинт `/api/health` для мониторинга состояния приложения и базы данных.
- **Мониторинг ошибок**: Интеграция с GlitchTip для отслеживания ошибок в production.

## Технологический стек

- **Язык**: Python 3.11+
- **Фреймворк**: [FastAPI](https://fastapi.tiangolo.com/)
- **База данных**: PostgreSQL
- **ORM**: SQLAlchemy 2.0 (асинхронный режим)
- **Миграции**: Alembic (подготовлено в зависимостях)
- **Валидация данных**: Pydantic v2
- **HTTP Клиент**: HTTPX (асинхронный)
- **Менеджер пакетов**: [uv](https://github.com/astral-sh/uv)
- **Контейнеризация**: Docker

## Быстрый старт

### Предварительные требования

- Установленный `uv` (рекомендуется) или `pip`.
- PostgreSQL.

### Установка

1. Клонируйте репозиторий:
   ```bash
   git clone <repository-url>
   cd events-provider-api
   ```

2. Создайте виртуальное окружение и установите зависимости:
   ```bash
   uv sync
   ```

3. Настройте переменные окружения:
   Создайте файл `.env` на основе `.env.example`:
   ```bash
   cp .env.example .env
   ```

## API Документация

После запуска приложения документация доступна по адресам:
- **Swagger UI**: [http://localhost:8000/docs](http://localhost:8000/docs)
- **ReDoc**: [http://localhost:8000/redoc](http://localhost:8000/redoc)

### Основные эндпоинты

- `GET /api/events` — Список мероприятий (пагинация: `page`, `page_size`; фильтр: `date_from`).
- `GET /api/events/{event_id}` — Детальная информация о конкретном мероприятии.
- `GET /api/events/{event_id}/seats` — Список доступных мест.
- `POST /api/tickets` — Регистрация на событие (бронирование билета). Поддерживает опциональный параметр `idempotency_key` для предотвращения дубликатов.
- `DELETE /api/tickets/{ticket_id}` — Отмена регистрации (удаление билета).
- `POST /api/sync/trigger` — Ручной запуск фоновой синхронизации.
- `GET /api/sync/status` — Текущий статус синхронизации.

### Статусы событий (`EventStatus`)

В API и модели БД статусы событий типизированы enum:

- `new`
- `active`
- `published`
- `registration_closed`
- `finished`

### Идемпотентность

При создании билета можно передать опциональный параметр `idempotency_key`. Это позволяет безопасно повторять запрос при ошибках сети без создания дубликатов:

- Если ключ уже использовался — возвращается существующий билет (HTTP 201).
- Если ключ использовался с другими данными — возвращается HTTP 409 Conflict.
- При ошибке провайдера результат не сохраняется — повторный запрос может быть обработан как новый.

### Outbox и уведомления

- После успешного создания билета в таблицу `outbox` сохраняется событие `ticket_created`.
- Отдельный воркер в фоне отправляет уведомление в Capashino и помечает запись как `sent` только после успешного ответа.
- Для ответов Capashino `5xx` включены быстрые ретраи в рамках одного цикла воркера (`OUTBOX_SEND_RETRIES_PER_CYCLE`), затем — отложенные повторы до `OUTBOX_MAX_RETRIES`.
- Для ошибок `400/401/422` запись помечается как `failed` без повторов (ошибки считаются непереисправимыми).

### Примечание по регистрации билетов

При высокой параллельной нагрузке место может стать недоступным между `GET /api/events/{event_id}/seats` и `POST /api/tickets`.
Сервис автоматически делает повторную попытку регистрации на следующем доступном месте перед возвратом `400 Seat is not available`.


- `Notification sent for ticket: ...`
